# 變更影響範圍分析（Context Map）

當使用者準備進行大型重構、介面修改、或跨多個模組的變更時，請先自動執行以下分析步驟，**確認影響範圍後再動手修改**。

## 觸發時機

以下情境應自動觸發此 Skill：

- 準備修改 `interface`、`abstract class`、`base class` 的成員簽名。
- 準備移動或重新命名檔案、類別、命名空間。
- 準備變更被多處引用的共用 DTO、Entity 或常數。
- 使用者說「重構」、「移植」、「拆分」、「合併」等關鍵動詞，且範圍跨兩個以上類別。

## 分析步驟

### 1. 識別變更核心

明確指出此次變更的「核心目標」是什麼（例如：「將 `IUserRepository.GetById(int)` 方法簽名改為接受 `Guid`」）。

### 2. 建立受影響清單

走查專案，列出：

- **直接實作者 / 繼承者**：實作目標 Interface 或繼承目標 Class 的所有類別。
- **直接呼叫者**：呼叫目標方法的所有位置（包含測試程式碼）。
- **間接依賴者**：依賴呼叫者的上層類別（若變更可能逐層傳遞）。

輸出格式：

```markdown
## 變更核心
[目標描述]

## 受影響範圍
| 類別 / 檔案 | 關係 | 需要調整的內容 |
| --- | --- | --- |
| ... | 直接實作 | 修改方法簽名 |
| ... | 呼叫端 | 更新傳入參數型別 |
| ... | 測試 | 更新 Mock 設定與測試資料 |
```

### 3. 確認後才執行

呈現分析結果後，**等待使用者確認影響範圍無誤**，再開始進行實際程式碼修改。

## 限制

- 若無法靜態走查（例如反射呼叫或動態代理），需標示「需手動確認」。
- 不強制在小型、顯而易見的修改中執行（如單一方法的內部邏輯調整）。
