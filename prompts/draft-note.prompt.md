# 整理筆記草稿

將使用者提供的討論素材、原始筆記或鬆散草稿，整理為內容結構完整的技術筆記。可在寫作過程中反覆執行。

## 輸入說明

使用者會提供以下任一形式的素材：

- 與 AI 討論後的對話摘要或重點整理。
- 尚未成形的草稿片段。
- 已有內容但結構凌亂的文件。

若素材中有明顯的技術主題，直接以該主題為核心整理；若主題不明確，先確認使用者的核心想表達的觀念，再開始整理。

## 輸出格式

### Frontmatter

```yaml
---
title: ""
date: YYYY-MM-DD
lastmod: YYYY-MM-DD
description: ""
tags: []
---
```

- `date` 與 `lastmod` 填入今天日期。
- `title`、`description`、`tags` 留空，待完稿後處理。

### 文章主體

1. **標題（H1）**：依核心主題暫定，供寫作期間識別用，完稿時再確認最終標題。

2. **開頭段落**：以個人敘事切入，說明這個主題怎麼浮現（如踩雷、同事分享、閱讀筆記拆分等）。語氣第一人稱、口語自然，不直接從定義開始。

3. **技術內容**：
   - 依邏輯順序拆分章節（H2/H3）。
   - 程式碼範例保留完整，不省略。
   - 適當使用表格整理對比資訊。
   - 若有需要補充說明的資訊，使用 VitePress 的 `::: tip` / `::: warning` 區塊。

4. **`## 異動歷程`**（固定章節，必須存在）：

   ```markdown
   ## 異動歷程

   - YYYY-MM-DD 初版文件建立。
   ```

## 風格要求

- 保留原始素材中的個人觀點與語氣，不過度學術化。
- 若素材中有「踩雷」、「同事問題」、「閱讀心得」等個人敘事元素，必須保留並融入開頭。
- 技術術語使用英文，說明文字使用繁體中文（台灣用語）。
- 不加入素材中沒有的技術觀點，若發現明顯資訊缺口，以 `⚠️ 待補：` 標記，不自行補充。

## 分工說明

| 工具 | 職責 |
| --- | --- |
| `draft-note` | 素材 → 內容結構（可反覆執行） |
| `proofread-note` | 完稿 → 潤飾可讀性 |
| `suggest-log-tags` | 完稿 → 確認 tags 與同分類文章一致 |
| `rename-log` | 完稿 → 確認標題、重新命名、搬移圖片 |
